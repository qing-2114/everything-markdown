# AGENTS.md

适用范围：本文件位于项目根目录，规则适用于整个 `everything-markdown` 项目。

## 基本要求

- 始终优先保持代码简单、清晰、可维护。
- 修改代码前先阅读相关模块，遵循项目现有结构和风格。
- 只修改完成当前需求所必需的文件，不顺手重构无关代码。
- 新增或修改逻辑后，尽量补充或更新对应测试。
- 回复和文档默认使用中文，除非用户明确要求英文。

## 代码规范

- JavaScript 使用现有 CommonJS 风格，保持函数职责单一。
- 渲染层、主进程、工具函数的边界要清楚，避免把 UI 状态、IPC、文件系统逻辑混在一起。
- 用户可见文案要集中管理，避免同一含义在多个位置重复硬编码。
- 路径、文件名、外部进程参数等必须通过结构化 API 处理，不手写拼接命令。
- 修改 Electron IPC、文件读写、卸载、外部进程调用时，要额外关注安全边界。

## 版本与安装包命名

- 每次更新代码并准备生成安装包时，必须同步更新 `package.json` 中的 `version`。
- 每次完成代码更新、修复或可运行功能变更后，必须自动执行 `npm run dist:win` 重新生成 Windows 安装包。
- 打包完成后必须检查 `release/` 中是否生成与当前版本一致的安装包，例如 `Everything Markdown Setup 0.1.3.exe`。
- 每次生成新安装包后，必须删除 `release/` 中旧版本的 `Everything Markdown Setup *.exe` 和对应 `.exe.blockmap`，只保留当前版本安装包及其 blockmap。
- 删除旧安装包前必须先确认当前 `package.json` 版本号和当前版本安装包文件都存在，避免误删最新产物。
- 如果打包失败，必须明确说明失败原因和下一步，不要声称安装包已更新。
- 版本号采用 `0.x.y` 递增规则：
  - 当前 `0.1.0` 更新一次后变为 `0.1.1`。
  - `0.1.9` 的下一次更新变为 `0.2.0`。
  - 依此类推。
- 安装包文件名使用 `package.json` 的版本号生成，因此版本更新后安装包名也会自动变化。
- 如果只改文档且不准备发布安装包，可以不更新版本号，也可以不重新打包；如果用户要求发布、打包或“更新安装包”，必须更新版本并重新打包。

## 并行处理

- 修改代码时，尽量把互不依赖的阅读、搜索、检查和测试并行执行。
- 并行只用于安全的只读操作或相互独立的验证任务。
- 文件编辑、格式化、提交等会改变工作区状态的操作必须谨慎串行执行。

## Superpowers Skills

- 本项目中处理需求时，默认自动使用适用的 superpowers skills。
- 如果用户明确要求不使用某个 skill，则以用户最新要求为准。
- 使用 skill 前要先读取对应说明，并按其流程执行。

## Git 与验证

- 开始代码工作前确认当前目录是 Git 仓库。
- 如果工作区有未提交改动，先按用户要求备份提交，再继续修改。
- 完成修改后先运行与改动相关的最小验证。
- 验证通过后先询问用户是否满意；用户确认后再提交。
- Commit message 必须使用中文，格式清晰。
